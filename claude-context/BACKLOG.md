## 🔴 Безопасность — приоритет

### 3. `helmet` отсутствует

**Что делать:** `npm i helmet` → `app.use(helmet({...}))` сразу после `cors`. Подобрать настройки CSP/COOP/COEP — сейчас COOP/COEP вручную, перенести в helmet.

### 4. Rate-limit на публичные write-эндпоинты

**Симптом:** `POST /api/orders`, `POST /api/checkout/cart` без троттлинга — можно спамить заказами / завалить БД.

**Что делать:** отдельный `rateLimit` (например, 20/мин/IP) на эти endpoints. Можно общий `rateLimit` на `/api` с более щедрыми лимитами + точечный жёсткий на write.

### 5. `cleanupUnusedImages` — деструктивная операция без страховки

**Симптом:** контроллер удаляет с диска всё, чего нет в строковых полях БД, начинающихся с `/uploads/`. Проблемы:
- Если URL — в JSON-массиве, в HTML richtext-описания товара (`<img src="/uploads/...">`), или абсолютный URL — файл не попадёт в `usedUrls` и **будет удалён**.
- Гонка: файл, загруженный между чтением БД и `unlink`, удалится как «неиспользуемый».
- Один POST стирает файлы без подтверждения.

**Что делать:**
- Парсить также HTML/JSON в richtext-полях (вытаскивать все `/uploads/...` через regex).
- По умолчанию — dry-run (`?dryRun=true` или возврат списка без удаления), `confirm=true` для реального удаления.
- Логировать каждое удаление с возможностью восстановления (или мягкое удаление — перемещение в `uploads/.trash/`).

### 6. Слой валидации схем (express-validator / zod / joi)

Чтобы не валидировать всё руками в каждом контроллере. Подключить один из, переписать `createOrder`, `updateOrderStatus`, `uploadRoutes` на него.

## 🟠 Логика / надёжность

### 7. `express.json({ limit: '30mb' })` — слишком много

API-телам в этом проекте достаточно ~100 КБ. 30 МБ — DoS-вектор. Снизить до 256 КБ глобально; если где-то нужно больше — точечно.

### 8. CORS пускает запросы без Origin (`!origin`)

Сейчас `whitelist.indexOf(origin) !== -1 || !origin`. Это пропускает curl и серверные запросы. Для прод-API стоит убрать `|| !origin` либо сделать это явным dev-режимом.

В whitelist также есть `http://` origin'ы (`localhost`, IP) — почистить когда переедет на финальный домен.

### 9. Кросс-доменные куки сессии

Фронт на Vercel (другой домен), кука сейчас `sameSite: 'lax'`. Для кросс-доменных POST/cookie-запросов с фронта (загрузка из админки, смена статуса) понадобится `sameSite: 'none'` + `secure: true` + `app.set('trust proxy', 1)` если за reverse proxy, плюс CORS `credentials: true` (уже есть) и точные origin'ы (не `*`).

**Проверка под прод-домен:** залогиниться в `/admin`, дёрнуть API из браузера на проддомене — кука должна уходить.

### 10. Логи с PII (email + IP) на каждый логин

`server.js:59`. Для прод-окружения — пересмотреть (GDPR, retention). Минимум — писать в файл/syslog с ротацией, не в stdout навсегда.

### 11. Graceful shutdown

На `SIGTERM` сейчас процесс падает резко. Нужно: закрыть HTTP-сервер, дождаться активных запросов, закрыть `sequelize` и `sessionStore`. В Docker важно для чистого рестарта.

## 🟡 Продукт / отложенное

### 12. Email-уведомления покупателю

`src/services/emailService.js` — заглушка, не вызывается. По продуктовой
модели покупатель получает письма о смене статуса (принят → в работе →
готов/отправлен). Когда будет SMTP:

- Реализовать `sendOrderStatusUpdateEmail` через nodemailer.
- Вызвать его из `updateOrderStatus` **после успешного `commit`** (не до — иначе письмо при откате транзакции).
- Без блокировки ответа: либо `await` с try/catch (письмо не должно ломать ответ), либо очередь/setImmediate.

### 13. `errorHandler` отдаёт `err.message` клиенту

Сейчас бизнес-ошибки уходят дословно: `"Not enough stock for product <name_ru>..."`. Утечка внутренних деталей. Опционально: типизированные коды ошибок, наружу — обобщённые сообщения, детали в лог.
